# Production Math-Reasoning Prompts: Multi-Agent Research Dive v1

**Date**: 2026-05-09
**Author**: Rain (with multi-agent synthesis)
**Status**: Active reference document
**Related**: `experiments.md` (Kaggle SP26 v2 OpenR1 training)

## Purpose

This document captures findings from a multi-agent research dive into production math-reasoning prompts used by competitive AIMO teams, open-source reasoning model labs, and community practitioners. The triggering question was practical: *what should the inference-time system + user prompt look like for a Qwen3-4B-Thinking-2507 SFT-finetuned model competing on a Kaggle math reasoning competition (mixed MCQ + free-form)?*

The dive does not directly answer that question. Instead, it surfaces the patterns top performers use, the contested space (length control, position effects), and an empirically-grounded prompt corpus to draw from when running prompt sweeps later.

## Methodology

Three external LLM agents (ChatGPT, Gemini, fresh Claude session with web search) were asked the same brief: surface verbatim production prompts from real competition writeups, model cards, GitHub repos, and community discussions. Each agent operated independently. Their outputs were then triangulated — Claims appearing across multiple agents with verifiable URL sources received high reliability tags; single-source claims received lower tags; contested claims (where agents disagreed) were called out explicitly.

Augmented with first-author web search for verification of high-impact claims (Qwen3 vendor recommendations, NemoSkills paper text, Fast-Math-R1 model card).

## Key takeaways (TL;DR)

1. **Production math-reasoning prompts converge on minimal user-only directives.** Top AIMO 2 teams used 1-2 sentence user prompts with no system prompt or only a generic system message. The "complex prompt engineering" instinct does not survive contact with reasoning models.

2. **System and user content function as a pairing, not as separate variables.** Where a particular instruction lives ("reason step by step" in system vs. user) varies between teams. What matters is the *total instruction landscape*, not where each piece is positioned. This rules out "design the user prompt" as a useful research question — the better question is "design the prompt pairing as a unit."

3. **Length-control via prompt is contested and largely doesn't work.** Multiple sources show explicit brevity directives ("think briefly", "minimal reasoning steps") backfire — Nemotron tests showed 21% MORE thinking after such instructions. Top teams achieve brevity through training-time interventions (DPO with length-aware reward, GRPO format/cosine/length rewards) rather than prompts.

4. **Qwen3-Thinking-2507 has specific vendor recommendations** worth treating as defaults: Temperature=0.6, TopP=0.95, TopK=20, MinP=0; output budget 81,920 for math/programming; for MCQ specifically, format `"Please show your choice in the answer field with only the choice letter, e.g., \"answer\": \"C\"."` (verbatim from model card).

5. **DeepSeek-R1 explicitly recommends NO system prompt.** "All instructions should be contained within the user prompt." This is meaningful because R1 was the architectural template for Qwen3-Thinking and influenced its training distribution.

## Cluster framework

Production math-reasoning prompts cluster into eight distinguishable patterns:

### Cluster A — Minimal user-only directive (DOMINANT)
1-2 sentences in user prompt only. Specifies: solve the problem, produce reasoning, put answer in \boxed{}. Used by NemoSkills (1st AIMO 2), Fast-Math-R1 (9th AIMO 2), Qwen3 official examples. **Reliability: high.** This is what top teams do.

### Cluster B — Generic system + minimal user
Bland system message ("You are a helpful and harmless assistant. You should think step-by-step.") + minimal user instruction with `\boxed{}` directive. Used by AceReason-1.1-7B and variants. **Reliability: medium-high.** Common in HuggingFace-published reasoning model variants.

### Cluster C — Heavy system, minimal user
Long, structured system prompts (sometimes 1000+ words specifying domain, format rules, style guidelines) with minimal user content. Used by OREAL. **Reliability: low-medium.** Outlier-ish; OREAL is one team's specific approach, not a convergent pattern.

### Cluster D — Domain-restricted system
System prompt explicitly bounds the model's domain ("You are an expert competition mathematician..."). Adds problem-domain framing. Used by some Astral-4B variants and academic reasoning models. **Reliability: medium.** Appears in writeups but does not clearly outperform Cluster A in head-to-head comparisons.

### Cluster E — Format-strict
Heavy emphasis on output format with explicit instructions about answer placement, multi-part answer handling, decimal/fraction conventions. Often combined with Cluster A's minimal user directive. **Reliability: medium-high** for production use; the emphasis varies between teams but the strict-format orientation is consistent.

### Cluster F — Length-controlled (CONTESTED)
Explicit instructions to limit reasoning length: "think briefly", "minimal reasoning steps", "be concise". **Reliability: low.** Multiple sources show this *backfires* — model produces *more* thinking, not less, when instructed to be brief. Top teams achieve brevity through training, not prompts.

### Cluster G — MCQ-specific format
Specialized prompts for multiple-choice, with explicit format directives like Qwen3-Thinking's recommended `"Please show your choice in the answer field with only the choice letter, e.g., 'answer': 'C'."`. **Reliability: high** when matched to MCQ-only inputs. Mixed-format competitions (MCQ + free-form) need a different approach (see open questions below).

### Cluster H — Idiosyncratic / outliers
Star Trek prompts, character-roleplay framings, "you are a math elf" — these exist in writeups but are not convergent. Worth knowing they exist; not for production. **Reliability: low.** Treat as curiosities.

### Cluster E (format-strict, often layered with A)

**E1. imagination-research (2nd AIMO)**
> System: "You are a helpful math assistant. Please reason step by step to put the answer in \boxed{}."
>
> User: "You excel at reasoning. You must put the final answer in \boxed{}. If the final answer is greater than 1000, then take the modulo of 1000. Think carefully and thoroughly, avoid duplication."

Source: imagination-research GitHub and AIMO 2 writeup. **Reliability: high.** 2nd place finisher. Note: the modulo-1000 instruction is *competition-specific* (AIMO 2 had this rule); not transferable.

**E2. Multi-part answer handling**
> System: (Cluster A or B style)
>
> User: "Solve the problem step by step. If the problem has multiple sub-answers, separate them by commas inside a single \boxed{}, e.g. \boxed{3, 7}. Give numerical answers to at least 4 significant figures, unless the problem specifies a different precision."

Source: composite from various format-strict writeups + Rain's current SP26 v1-baseline prompt. **Reliability: medium-high** for the format directive itself; **note for Rain**: the "4 significant figures" directive may be problematic given Piazza answer-format mismatch concerns (see `experiments.md`).

### Cluster F (length-controlled — contested)

**F1. Brevity-directive (BACKFIRES)**
> User: "Solve the problem briefly. Provide minimal reasoning steps. Put the answer in \boxed{}."

Source: Nemotron team's reported test of this pattern. **Reliability: high evidence that this pattern produces *more* thinking, not less.** Reported 21% increase in reasoning length when this directive is added. Translation: explicit length instructions in math prompts are counterproductive.

**F2. Working approach (training-side)**
> Approach: GRPO with format reward, cosine reward (preferring brief correct answers), and length reward.

Source: Fast-Math-R1 training methodology. **Reliability: high.** This is what works when you actually want shorter reasoning at inference — train it in, don't prompt it.

### Cluster G (MCQ-specific)

**G1. Qwen3-Thinking-2507 official MCQ**
> User: (problem + answer choices) + "Please show your choice in the answer field with only the choice letter, e.g., \"answer\": \"C\"."

Source: Qwen3-Thinking-2507 model card and Qwen3 documentation. **Reliability: very high.** Vendor recommendation specifically for MCQ tasks.

**G2. DeepSeek-style MCQ**
> User: (problem + choices) + "Reason through the options and select the best one. Output your final answer as a single letter."

Source: various DeepSeek-derived community examples. **Reliability: medium.** Not vendor-specified for R1.

## Contested findings

### Length control: prompt vs. training

**The contested claim**: Can you make a reasoning model think less by telling it to think less?

**The evidence**:
- Multiple sources (Nemotron tests, plus several anecdotal reports) show explicit brevity directives produce *more* thinking, not less.
- Top teams (Fast-Math-R1) achieve brevity through training-time interventions, not prompts.
- **Verdict**: Length control via prompt does not work. Plan for it via training (DPO, GRPO with length-aware rewards) if you need it.

**Implication for Rain**: Don't add brevity instructions to the prompt. The current prompt is silent on length, which is the right default.

### System vs. user position effects

**The contested claim**: Does the same instruction (e.g., "reason step by step") produce different results when placed in system vs. user?

**The evidence**:
- The dive found **no controlled comparisons** that isolate position from content.
- Different teams put different instructions in different positions, but no team published "we A/B tested system vs user with the same content."
- **Verdict**: Unknown. Worth running ourselves once we have a winning model. See "Open questions" below.

**Implication for Rain**: For the v2 OpenR1 inference, hold prompt position constant. Defer this question to a controlled study after a winner is selected.

### Heavy system vs. minimal system

**The contested claim**: Do long structured system prompts (Cluster C, OREAL-style 1300 words) outperform minimal generic system prompts (Cluster B)?

**The evidence**:
- OREAL ships heavy system; AceReason ships minimal generic; both are competitive published reasoning models.
- No public head-to-head with the same training data.
- **Verdict**: Unclear. Top AIMO 2 teams used minimal system or no system. The 1st-place team (NemoSkills) used Cluster A (no system).

**Implication for Rain**: Default to Cluster A or B for v2 OpenR1 inference. Don't invest in a heavy system prompt without evidence it helps.

## Methodology notes (recurring across dives)

These are observations about *how* the research dive was conducted that may be useful for future dives.

1. **Triangulation works.** Three independent agents converged on Cluster A as the dominant pattern. The agreement is strong evidence the pattern is real, not artifact of one agent's training data.

2. **Disagreement is signal.** Where agents disagreed (length control, system vs. user position effects), the disagreement itself was useful — it pointed at unresolved questions.

3. **Contested claims need the source URL or it's just rumor.** Several agents made strong claims without verifiable sources. Those got demoted. Future dives: require URL evidence for any high-impact claim.

4. **Verbatim quotes outperform paraphrase.** When extracting prompts from sources, verbatim text is the safe move. Paraphrased prompts are essentially fictional unless the source uses the same words.

5. **Document the agents themselves.** What model version, what date, what tools they had. Future-Rain reading this will need to know whether the source was a 2024 GPT-4 or a 2026 GPT-5; the difference matters.

## Implications for v2 OpenR1 inference (immediate)

This dive does not directly choose Rain's inference prompt. The decision is: **hold the v1-baseline prompt constant across all three v2 arms (NuminaMath, OpenR1, Frugal). Defer the prompt sweep to the chosen winner.**

Rationale: variance from prompt changes confounds variance from training-data changes. Single-variable testing requires holding the prompt fixed during training comparisons.

**For the prompt sweep on the winner**: this corpus is the starting set. Suggested order:
1. Cluster A baseline (NemoSkills minimal)
2. Cluster A + format-strict additions (current v1-baseline + the multi-part \boxed{} convention)
3. Cluster A with Qwen3 vendor sampling parameters (Temp=0.6, TopP=0.95, TopK=20)
4. MCQ-specific Cluster G prompt for MCQ-only test slice
5. Anything from Follow-up 3's MCQ patterns dive that surfaces

## Open methodology question

For Rain's research project (1 month away): **how should anecdotal evidence from competition writeups, model cards, and forum discussions be cited and weighted in research output?** This dive uses URLs as the standard — every high-impact claim has a verifiable source. But "verifiable" doesn't mean "rigorous" — most of these sources are first-person reports, not controlled studies. The reliability tags are subjective. Future research dives should be honest about this: what we know is not "proven" but "what people report."
