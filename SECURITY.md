# SECURITY.md — Credentials and Secrets

## RULES (revised 2026-05-29 — supersedes 2026-05-28 lock)

1. **NEVER put secrets in this repo.** Not in code, not in docs, not in committed prompt files, not in markdown examples. Git history is permanent.
2. **NEVER embed a PAT, API key, password, or any credential in a spawn prompt or an agent-to-agent message.** This is the specific failure mode from the 2026-05-28 incident — embedding a token in a prompt that gets handed to multiple runtimes multiplies the exposure surface (chat histories, agent execution traces, screen captures). Hard prohibition.
3. **Rain MAY provide a PAT directly to a Claude in chat for that Claude's runtime.** Token requirements:
   - **Fine-grained, repo-scoped** only — never classic `ghp_` tokens (they over-scope)
   - **≤7-day expiry**
   - Rotate within 24 hours of session end
4. **When a Claude receives a PAT via chat**: write to `~/.git-credentials`, use for the session, NEVER re-display the token, NEVER embed it in subsequent outputs.
5. **Any token disclosed in chat is burned at session end.** Rotate.
6. **If an agent hits a git auth error**: REPORT it to Rain. Do not ask another Claude or another agent for a PAT — only Rain provisions credentials.

## One-time setup (run on each runtime that needs write access)

```bash
git config --global credential.helper store
echo "https://dvaneetv:YOUR_FINE_GRAINED_PAT@github.com" > ~/.git-credentials
chmod 600 ~/.git-credentials
git config --global user.email "dvaneetv@ucsd.edu"
git config --global user.name "claude_agent"
```

For chat-based Claudes (ephemeral sandboxes), Rain re-runs this at the start of each session that needs push capability.

## Spawn prompt SETUP block (the only permissible form for agent-to-agent setup)

```
SETUP:
  git clone https://github.com/beepbeeepimajeep/151B_SP26_Competition.git /home/claude/repo
  cd /home/claude/repo
  Credentials are pre-configured in ~/.git-credentials by Rain (or by Rain
  having run the setup at the start of this runtime). If git push fails with
  auth error, REPORT to Rain — do NOT request a PAT in chat from any other
  agent.
```

The spawn block NEVER contains the actual token. This is the rule the 2026-05-28 incident violated.

## When something leaks

1. Immediately revoke at GitHub → Settings → Developer settings → Personal access tokens → Revoke
2. Generate a replacement (fine-grained, ≤7-day, repo-scoped)
3. Re-run the one-time setup on each runtime that needs write
4. Log the incident in the "Past incidents" section below

## Past incidents

- **2026-05-28**: classic PAT `ghp_E2zO...` (first 8 chars only, full token revoked) was embedded in multiple spawn prompts generated by claude_strategy. Exposed in chat history and agent execution traces, not in any commit. Revoked and rotated. The over-broad "never accept PATs in chat" rule was locked in response — see 2026-05-29 amendment for why that was over-corrected.
- **2026-05-29 (a)**: Rain pasted a token starting with `ghp_E2zO...` in chat (potentially a re-paste of the 2026-05-28 revoked token, or a new classic PAT — the prefix collision is suspicious). Treated as burned regardless. claude_strategy refused to consume it under the then-locked rule.
- **2026-05-29 (b)**: Policy revised in light of operational reality — ephemeral chat sandboxes have no out-of-band credential provisioning path, and the 2026-05-28 lesson was specifically about agent-to-agent embedding (which remains locked), not about Rain → Claude direct provision for a single session.
